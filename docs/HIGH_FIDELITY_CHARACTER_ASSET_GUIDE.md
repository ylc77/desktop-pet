# 高保真角色素材制作规范

本文档规定未来正式桌宠角色的素材制作与交付要求。它不授权开始制作任何具体角色，也不代表当前中性几何占位角色已经达到正式美术质量。

## 当前技术边界

- 当前 Release 唯一正式验证路径是 `schemaVersion: 1` 角色包中的 RGBA 透明 PNG 序列帧。
- 正式素材必须作为独立角色资源包接入，不能把角色名称、帧数、动作路径或外观规则写入 React、Rust 等核心业务代码。
- 外观系统不限定人物、动物、机器人、幻想生物、真人、宠物、游戏题材或某种固定画风。每个角色以该次制作确认的设定、参考权利和 production profile 为准。
- 框架已在 `schemaVersion: 1` 内提供可选的前摇/收尾、环境动作持续时间、移动和独立阴影元数据；旧角色包无需增加这些字段。若未来再增加元数据，仍必须保持旧角色包完全可用，并同步实现文档、校验器与自动测试。
- `app-icon.png` 中的白猫照片目前只用于七酱桌宠的应用图标。未经用户明确确认，不得把照片转换为桌宠角色、作为角色外观依据，或据此选择猫咪、写实、卡通等风格。
- 正式角色制作开始前，必须先确认角色类型、参考对象及授权、画风、身体比例、写实程度、配色、性格、服装或动物特征、首批动作和动画形式；帧数、FPS 与动作阶段由 Codex 按本角色实际复杂度提出最优方案并写入 production profile 供用户确认。

技术系统通过自动测试，只表示它具备承载素材的能力。正式角色是否自然、统一、有表现力，必须另行完成人工视觉验收。

## 每个角色先建立 production profile

开始绘制或生成动作帧前，由 Codex 与用户共同填写 [角色制作规格模板](CHARACTER_PRODUCTION_PROFILE_TEMPLATE.md)。该 profile 是制作和验收记录，不是新的 manifest 字段，也不提升 `schemaVersion`。至少锁定：

- 角色类型、画风、写实程度、目标显示高度、画布与锚点。
- 正面/侧面或左右方向策略，不对称元素能否镜像。
- 每个动作的交互目的、复杂度、关键姿势、实际帧数、FPS、循环和前摇/收尾。
- 目标包体、估算 RGBA 解码占用、测试设备和质量/性能取舍。
- 公开或私人用途、参考来源、肖像同意和允许的发布范围。

项目不设置一套全局固定帧数或 FPS。Codex 应在每次角色制作时结合目标逻辑像素尺寸、轮廓与材质细节、动作持续时间、关键姿势数量、移动速度和 64 MiB 左右的运行时解码引用预算提出方案。优先保证关键姿势和节奏，再选择能自然呈现动作的最低有效帧数与 FPS；不得用无意义补间堆高帧数，也不得为了压缩包体删掉动作可读性所需的前摇或收尾。

服装、发型、装备或完整配色发生显著变化时，第一版按独立角色包和独立 ID 制作、校验和发布。只有未来确立了跨所有动作都可靠的图层对齐与组合协议后，才考虑共享皮肤层；当前不得把不同服装强行塞进同一套帧或硬编码到业务逻辑。

## 交付结构

正式角色沿用现有资源包结构：

```text
public/characters/<character-id>/
  manifest.json
  preview.png
  animations/
    idle/
      idle_0001.png
      idle_0002.png
    walk_left/
    walk_right/
    sleep/
    hover/
    click/
    double_click/
    drag/
  metadata/
    source.md
    license.md
```

需要交付给用户导入时，运行 `npm run package:character -- -CharacterId <角色ID>` 生成 ZIP 兼容的 `.qipet`。包根直接包含上述角色内容而不是外层角色目录，具体流程见 [角色资源包制作说明](CHARACTER_PACK.md)。

动作目录按最终确认的动作清单提供；除 `idle` 外，示例动作并非全部强制。文件命名、尺寸限制、FPS 范围和 manifest 规则以 [角色资源包制作说明](CHARACTER_PACK.md) 为准。

## 硬性画面规范

### 画布、比例与构图

1. 同一角色包的所有帧必须使用与 manifest 一致的固定画布尺寸。
2. 所有动作中的角色比例必须一致，不能因动作切换出现整体变大或缩小。
3. 角色实际占用尺寸与四周透明留白应保持统一；不得通过逐帧自动裁切改变构图。
4. 脚底或身体基准锚点必须稳定。站立、行走、交互动作切换时，不应出现非动作意图造成的跳位。
5. 摄像机视角、透视和观察高度必须一致。只有经过设计确认的转身或特殊镜头动作可以改变视角。
6. 左右移动素材应在相同视觉尺度和接地基准下制作。

### 造型连续性

1. 服装、配色、纹理、饰品和标志性细节必须跨帧、跨动作保持一致。
2. 不得随机增加、减少、替换或移动饰品。
3. 手指、耳朵、角、翅膀、尾巴等结构的数量和连接关系不得变化。
4. 发型、脸型、五官、花纹和其他身份特征不得漂移。
5. 衣物、头发、耳朵或尾巴可以有次级运动，但其长度、形状和固定点必须连续。
6. 左右方向不得直接镜像带有文字、徽记、单侧发饰、单侧服装结构或其他方向不对称细节的画面；应提供正确绘制的方向素材，或经人工确认镜像不会产生错误。

### 光影与透明度

1. 主光源方向、明暗关系和材质表现必须保持一致。
2. 透明边缘必须干净，不得有白边、黑边、色边、锯齿污染、脏像素或半透明底色。
3. 输出必须保留真正透明背景，不能把棋盘格或纯色预览背景烘焙进图片。
4. 不要把只适合某个地面、方向或姿态的阴影烘焙进所有角色帧。
5. 若使用脚底接触阴影，应由程序的独立可禁用视觉层承载，或仅在经过逐动作确认确实适合时交付；不得导致透明窗口出现方形底色。

## 动作质量规范

### 动作结构

每个关键动作应根据需要包含：

1. 前摇（Anticipation）：为动作方向和力度建立预期。
2. 主体（Action）：清晰表达动作意图。
3. 收尾（Recovery）：消解惯性并自然进入后续状态。

短动作不要求机械地平均分成三段，但不得从静止瞬间跳到极端姿势，再无过渡地切回 `idle`。动作首尾应与 `returnTo` 对应状态在姿势、锚点、重心和表情上可自然衔接。

### 节奏与生命感

- 行走步频必须与程序中的实际移动速度协调，脚底不能明显滑动。
- 起步、停止和转向应表现出合理的重心变化，不能每帧等速、等幅地机械摆动。
- 呼吸和眨眼不应始终固定同相同步；眨眼时机、呼吸起伏和小幅次级运动应形成自然节奏。
- 随机变化不能破坏角色一致性，也不能靠随机缩放、随机位置或随机增减细节制造“生动感”。
- 点击、悬停、拖动、落地、睡眠和唤醒动作应能从合理姿态进入，并在结束后回到合理状态。
- 动作循环点不得产生额外停顿、位置跳跃、亮度闪变或轮廓抽动。

## 动作间衔接

交付前应至少验证以下切换：

- `idle` 到左右行走，以及行走回到 `idle`。
- `idle` 到睡眠，以及睡眠后的唤醒。
- `idle` 到 `hover`、`click`、`double_click`，再回到指定状态。
- 拖动进入、拖动保持、释放后的恢复。
- 左右方向切换以及接近屏幕边缘时的停止或转向。

检查时同时关注角色比例、锚点、脚底位置、重心、光影、表情和轮廓。不能只检查单个动作循环是否能播放。

## 输出与自检流程

1. 先由 Codex 填写并让用户确认该角色的 production profile，再从统一角色设定和动作表开始制作，锁定画布、比例、锚点、视角与光源。
2. 导出 RGBA PNG，使用四位连续编号，例如 `idle_0001.png`。
3. 不逐帧自动裁切，不依赖操作系统目录顺序。
4. 使用叠帧或洋葱皮检查脚底锚点、轮廓抖动、比例漂移和透明边缘。
5. 以正常速度、`0.5x` 和 `0.25x` 检查循环点、动作结构和衔接。
6. 在透明背景、浅色背景和深色背景上分别检查边缘污染。
7. 对照 production profile 记录实际帧数、FPS、动作时长和估算解码占用；任何偏离都要写明原因。
8. 运行 `npm run validate:characters`，修复全部结构、命名、格式和范围错误。
9. 在开发者工具中查看画布边界、锚点、 hitbox、帧序号和目标 FPS。
10. 按 [角色视觉人工验收清单](CHARACTER_VISUAL_REVIEW_CHECKLIST.md) 完成人工审核并保留审核记录。
11. 需要本地导入交付时生成 `.qipet`，复核包根结构、文件大小和 SHA-256，不覆盖未知同名文件。

校验器通过不表示构图、造型、表演或版权已经通过；这些项目必须由有权确认角色方案的人进行人工审核。

## 来源、授权与版权

公开内置或对外分发的正式资源包不得包含未经授权的：

- 现有版权角色、其高度可识别的变体或仿冒设计。
- 商标、品牌标识、服装徽记、产品包装或受保护图案。
- 真人肖像、宠物照片、摄影作品或私人图片。
- 第三方字体、笔刷、纹理、模型、素材包、生成模型输出或训练参考中不允许再分发、修改或商业使用的内容。

交付前必须完成以下记录：

- `metadata/source.md`：逐项记录原创、委托、用户提供或第三方来源，以及原始文件或授权证据的可追溯位置。不得把敏感本机绝对路径写入公开文件。
- `metadata/license.md`：记录版权所有者、允许的发布范围、修改权、商业使用权、再分发权、署名要求和限制。
- 若参考真人或宠物，需确认照片权利、肖像或隐私同意以及角色化使用范围。
- 若素材由生成式工具辅助制作，仍须确认输入参考有权使用、输出允许用于计划中的发行方式，并由人工排除与现有角色近似或细节侵权风险。

“网络可见”“由用户上传”“AI 生成”“仅作参考”均不自动等于可以发布。授权不明确时，不得将素材纳入正式角色包或发布候选。

私人本地包可以只在用户设备上导入并标记为“私人/本地”，不会自动成为公开角色。该技术边界不授予任何第三方 IP、商标、照片或肖像许可：游戏角色即使由用户重新绘制也可能仍受原 IP 限制，真人应取得本人及照片权利人的许可，宠物照片也需确认摄影作品权利。私人包不得被应用上传、推荐、捆绑或自动转为公开包；一旦计划分享或发行，必须按公开包标准重新审核来源与许可。

## 白猫应用图标的特别边界

当前白猫照片已经被确认用于应用图标，但该确认不自动扩展到角色设计或动画制作。只有用户明确回答“是否基于当前白猫照片设计角色”，并确认照片权利和允许的改编范围后，才可以把它纳入角色方案。即便得到许可，也必须先确认画风、比例、写实程度、标志特征和动作清单，不能直接由图标推导正式桌宠。

## 验收结论

正式素材只有同时满足以下条件才能标记为“美术验收通过”：

- 资源包结构与技术校验通过。
- 逐帧和实际播放的人工视觉检查通过。
- 角色方案、动作表现和方向素材得到用户或指定美术负责人的确认。
- 来源、授权和发布范围记录完整。

仅完成前两项中的技术部分时，应标记为“技术可接入”或“待美术验收”，不得描述为“正式角色已完成”“生动真实”或“视觉效果已验收”。
