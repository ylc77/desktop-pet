最终实现截图与方案 2 源图已在同一比较输入中复核。菜单保持紧凑暖白 Fluent 方向，使用三处分隔线和七个操作项；信息顺序、图标语义、浅蓝首项高亮、柔和圆角与阴影、红色“退出”均与源图一致。实现按实际 `420 × 420` 桌宠窗口收紧到 `124 px`，并优先贴近离点击位置较远的窗口边缘，避免覆盖角色主体；这是桌面宠物小窗口约束下的密度调整，不改变设计层级。

## Focused region comparison evidence

无需额外裁切。菜单在原生 `420 × 420` 截图中为 `124 px` 宽，图标、文字、分隔线、圆角、阴影与焦点状态均可在 1:1 截图中清晰辨认；所有常态菜单文案完整显示，同一比较输入已经同时覆盖完整构图和菜单细节。

## Findings

- 无 P0、P1 或 P2 问题。
- 字体与排版：使用 Windows 系统中文字体栈，字号、行高和字重符合紧凑原生菜单密度；文案无截断或换行。
- 间距与布局：36 px 菜单项、统一图标列、三处分隔和 10 px 圆角形成稳定节奏；菜单贴边显示且未裁切，不再覆盖角色主体。
- 色彩与视觉 token：暖白表面、低对比边框、浅蓝交互高亮、深灰正文和克制的危险红与源图一致；键盘焦点保持可见。
- 图像与素材质量：使用本地 Microsoft Fluent UI System Icons SVG 资源；线条清晰、透明背景正确，没有 emoji、字符图标或 CSS 绘制替代品。
- 文案与内容：顺序为“暂停动画、隐藏桌宠、外观中心、设置、开机启动、检查更新、退出”；已移除“关于七酱桌宠”，退出文案已简化为“退出”。

## Comparison history

1. 第一轮比较发现 P2：鼠标打开菜单时，程序化初始焦点显示了完整蓝色键盘焦点环，比方案 2 的柔和浅蓝高亮更重。
2. 修复：新增输入方式状态。鼠标打开时仅显示浅蓝高亮；用户开始使用方向键、Tab、Home 或 End 后才显示清晰蓝色焦点环。
3. 第二轮比较：返修后的实际截图与方案 2 再次放入同一比较输入，P2 已消除；没有新增可操作的 P0/P1/P2 差异。
4. 用户实机反馈 P2：`184 px` 菜单仍覆盖角色右半侧，且点击桌面后菜单不会消失。
5. 修复：菜单进一步收紧到 `124 px`，根据点击点所在半区选择对侧窗口边缘；增加窗口失焦关闭，并保留菜单外背景点击关闭。
6. 最终比较：原生右键打开后菜单完整显示在角色主体右侧；切换焦点到其他桌面窗口后菜单立即收起。P2 已消除。
7. 用户手动测试发现 P1：菜单虽然可见，但透明窗口仍只允许桌宠命中区域接收鼠标，移动到菜单后点击可能穿透到桌面或后方窗口。
8. 修复：菜单打开时暂时发布“整窗可交互”状态；菜单关闭后重新计算并恢复桌宠命中区域。视觉布局保持不变。
9. 修复后验证：新增回归测试覆盖“桌宠命中区 → 整窗交互 → 恢复桌宠命中区”，完整前端测试 `292/292` 通过；等待用户进行最终手动逐项点击验收。

## Primary interactions tested

- 右键桌宠：菜单正常打开并在窗口边缘内完整显示。
- 左键点击桌面或其他窗口：桌宠窗口失焦，菜单立即关闭。
- 菜单打开期间：整张菜单均处于可交互区域，不再把点击穿透给桌面或后方窗口。
- `ArrowDown`：焦点移动到下一项，并切换为清晰键盘焦点环。
- `Esc`：菜单关闭，桌宠继续运行。
- 单元测试已覆盖动作前先关闭菜单、项目顺序、移除旧文案和键盘循环导航。

## Console/runtime errors checked
