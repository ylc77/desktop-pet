# Updater 密钥管理

> 当前决策（2026-07-16）：本轮不生成生产更新密钥、不输入生产密码、不执行加密备份，也不操作远端 Release 或 endpoint。自动更新生产配置继续保持禁用。下文保留为安全参考，不代表密钥、备份或远端资源已经创建。

七酱桌宠的 Tauri Updater 使用 Ed25519 签名验证更新包。当前仓库只准备密钥初始化、验证和泄露扫描工具；**生产私钥尚未生成，正式公钥尚未配置**。

## 两类签名不能混淆

| 签名 | 用途 | 当前状态 |
|---|---|---|
| Tauri Updater 签名 | 让已安装应用验证更新包来自同一发布者且未被篡改 | 基础已接入，生产密钥未配置 |
| Windows Authenticode 代码签名 | 向 Windows 和 SmartScreen 提供可信发布者身份与文件签名 | `NotSigned` |

Updater 签名不能消除“未知发布者”或 SmartScreen 风险；Windows 代码签名也不能替代 Updater 的 `.sig` 验证。

## 生产密钥初始化前置条件

达到生成生产密钥的步骤时必须停止，并由项目所有者确认：

1. 私钥保存路径。建议默认位于仓库外：`%USERPROFILE%\.tauri\qijiang-desktop-pet.key`。
2. 由用户交互输入至少 16 个字符的生产密码；密码不得出现在命令行历史、脚本、`.env`、日志或报告中。
3. 至少两个彼此独立、受控且离线的备份位置已经准备好。

先只预览：

```powershell
.\scripts\updater\initialize-updater-key.ps1 -WhatIf
```

未经用户确认，不运行真实生成。初始化工具必须拒绝仓库内部路径、已有文件覆盖、少于 16 个字符的生产密码，以及私钥或公钥路径中任一级的符号链接、目录联接、挂载目录或其他 reparse point；不在终端显示私钥或密码。生成与配对验签先在唯一 staging 文件中完成，发布时先移动非秘密公钥、最后移动私钥；异常清理前再次核对本次生成文件的 SHA-256，不删除并发出现的未知文件。只有公钥正文可以进入构建配置。

> 丢失生产私钥后，已经安装旧公钥版本的用户无法验证由新私钥签署的更新。备份必须在首个公开版本发布前完成并验证可恢复。

## 双离线备份

密钥生成并验证后，使用专用工具把**加密私钥及其公钥**复制到两块独立的可移除介质。源密钥、备份一和备份二必须能够识别为三个不同的物理磁盘；两个备份目标还必须识别为 USB、SD、MMC 或 Windows 可移除驱动器。固定内置硬盘、虚拟磁盘、文件支持的虚拟磁盘、Storage Spaces、未知总线和网络存储不能算离线备份。同一物理盘上的两个盘符或两个目录也不能算两份独立备份。脚本无法可靠识别物理磁盘时会安全失败，不会用盘符不同来猜测通过。

先连接两块分别受控的 USB 存储介质，只执行预览：

```powershell
.\scripts\updater\backup-updater-key.ps1 `
  -KeyPath "$env:USERPROFILE\.tauri\qijiang-desktop-pet.key" `
  -BackupDirectoryOne 'E:\七酱桌宠 updater key backup' `
  -BackupDirectoryTwo 'G:\七酱桌宠 updater key backup' `
  -WhatIf
```

确认预览识别到两个不同物理磁盘后，移除 `-WhatIf` 执行。工具会：

- 拒绝仓库内源文件或备份目录。
- 验证私钥是预期的加密 Tauri 密钥容器，并验证公钥结构。
- 拒绝任何已有同名文件，不覆盖旧备份。
- 分别复制私钥和公钥，并对每个副本做 SHA-256 字节级核对。
- 只输出脱敏路径、介质类型、公钥指纹和成功/失败状态，不输出私钥正文、完整本机路径或密码。

物理磁盘识别使用 Windows Storage 管理接口。若受限会话返回“无法证明物理磁盘独立”，请连接好两块介质后在提升权限的 Windows PowerShell 中重新运行；不得通过关闭检查或仅比较盘符来绕过。符号链接、目录联接和挂载文件夹也会被拒绝，避免显示路径和实际写入磁盘不一致。

复制和哈希校验成功后，介质**仍处于在线状态**。必须在 Windows 中安全弹出或物理断开两块介质，分别保管，才可将其记录为两份离线备份。不要把两个目录建在同一 USB 盘，也不要把同步云盘目录当作已经离线的备份。

正式发布前还要做一次受控恢复演练：从每块已断开的介质分别读取副本，重新运行 `verify-updater-key.ps1 -VerifySigning`，确认密码可解锁、私钥可签名且公钥可验签。恢复演练不得覆盖原始密钥。

## 验证

密钥存在后使用验证脚本确认：

```powershell
.\scripts\updater\verify-updater-key.ps1 -KeyPath '<仓库外私钥路径>' -VerifySigning
```

验证范围应包括：

- 私钥路径在 Git 仓库外且未被跟踪。
- 密码通过安全交互或仅限当前进程的环境变量提供。
- 私钥可以签署临时测试文件，配置公钥会对该文件和生成的 `.sig` 做真实密码学验签；不能仅以 `.sig` 存在、公钥指纹相同或签名文本非空判定通过。
- 错误公钥和被修改的文件验证失败。
- 临时测试文件与测试密钥在结束后删除。

验证报告只能记录通过/失败、公钥指纹和脱敏路径，不得记录私钥、公钥之外的密钥材料、密码或完整用户目录。私钥与公钥的输入路径都必须拒绝 reparse point，避免校验对象与实际读取对象不一致。

## 构建时使用

官方签名构建只在当前进程内设置：

- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`

构建结束后立即清理，不写入 `.env`。生产密码至少 16 个字符，且不得作为 PowerShell 明文参数。`build-signed-update.ps1` 在签名前必须验证 HTTPS endpoint、公钥、版本一致性、工作区状态、私钥位置与所有密钥路径没有 reparse point；外部公钥在长构建前固化为唯一 snapshot，私钥字节在发布前重复核对。构建使用唯一的隔离 `CARGO_TARGET_DIR`，只接受精确产品名、版本和架构对应且由本次构建生成的安装包。复制到发布目录后还要用该公钥 snapshot 真实验证最终安装包的 `.sig`；验签失败不得留下可发布目录。

## Git 与泄露防护

仓库忽略私钥、PEM、密码文件、私钥备份目录和临时签名产物；正式公钥是公开验证材料，可以按构建流程纳入配置。运行：

```powershell
.\scripts\updater\scan-updater-secrets.ps1
```

扫描 tracked 文件、Git index 中 staged blob 的实际内容、Release、QA 和日志中的密钥环境变量名、私钥头部、默认生产私钥绝对路径与其他已知私钥路径；不能用工作区同名文件代替 staged 内容。超过 10 MiB 的候选文本不会被静默跳过，而是安全失败并报告无法完整扫描的位置。工具只能报告类别和文件位置，不输出匹配到的秘密正文、默认私钥完整路径或其他敏感值。

若怀疑泄露：立即停止发布，保留最小审计证据，判断是否已有旧公钥版本公开安装。不要直接覆盖或删除唯一密钥。密钥轮换必须设计兼容迁移；在没有旧私钥时，不能通过替换服务器文件让旧客户端信任新密钥。
