# 桌宠开发者项目需求与外观制作规范

> 项目定位：先开发一个稳定、可扩展、可打包为 Windows `.exe` 的通用桌宠框架。第一阶段只完成核心运行能力、动画系统、角色资源协议和开发工具；正式角色外观在框架稳定后制作。

---

## 1. 项目目标

开发一个可独立安装和运行的 Windows 桌宠应用。最终用户安装后，不需要打开 ChatGPT、Codex、浏览器、Node.js、Rust 或其他开发工具即可使用。

项目第一阶段的目标不是制作某个特定人物的专属功能，而是完成一个可重复使用的桌宠开发框架：

- 透明、无边框的桌宠窗口
- 鼠标拖动、点击和基础互动
- 可配置的动画状态机
- 独立的角色资源包系统
- 设置持久化
- 开机启动、系统托盘和退出控制
- Windows 安装程序
- 开发调试、资源校验和错误回退
- 后续可以添加任意角色、皮肤和扩展功能

---

## 2. 当前阶段范围

### 2.1 第一阶段要做的内容

- 通用桌宠窗口
- 动画播放与状态切换
- 鼠标交互
- 角色资源加载
- 角色切换
- 设置保存
- 系统托盘和右键菜单
- 开机启动
- 日志与调试工具
- Windows 构建和安装包
- 使用说明与开发文档

### 2.2 第一阶段暂不加入的内容

以下内容不属于当前核心版本，先不开发：

- 与桌宠运行框架无关的专属内容模块
- 日程、提醒和信息服务模块
- AI 聊天
- 在线语音识别和语音合成
- 天气、日历和待办事项
- 账号系统
- 云同步
- 商城、会员和支付
- 广告系统
- Live2D 或复杂骨骼动画
- 联机互动
- 大型小游戏

这些能力以后应作为独立插件或可选模块加入，不能与核心框架耦合。

## 3. 核心开发原则

### 3.1 核心功能优先

先用占位图片完成整个程序流程，再制作正式外观。

推荐顺序：

1. 创建透明窗口
2. 完成拖动、置顶和退出
3. 完成动画播放器
4. 完成动画状态机
5. 完成角色资源包协议
6. 完成设置保存
7. 完成开发调试工具
8. 完成安装包
9. 使用占位角色完成稳定性测试
10. 再确认正式角色风格并制作外观

### 3.2 功能层与美术层分离

核心代码不允许写死某个角色的：

- 名称
- 图片路径
- 帧数
- 帧率
- 动作列表
- 尺寸
- 锚点
- 气泡内容
- 音效

这些信息应放在角色资源包的配置文件中。

### 3.3 配置驱动

角色、动画和皮肤都通过配置文件加载。增加新角色时，应主要通过复制资源目录并修改配置完成，而不是修改核心程序。

### 3.4 离线优先

第一版的全部核心功能必须可以离线运行。网络功能以后作为可选扩展加入。

### 3.5 错误不能导致主程序崩溃

遇到以下情况时，程序应记录错误并回退：

- 某个动画目录不存在
- 某一帧图片损坏
- 配置字段缺失
- 角色资源包版本不兼容
- 当前显示器断开
- 保存的窗口位置已经无效

优先回退到占位角色或 `idle` 动画。

## 4. 推荐技术方案

### 4.1 应用技术栈

推荐：

- Tauri 2
- React
- TypeScript
- Rust
- Vite

### 4.2 第一版动画方案

优先选择透明 PNG 序列帧。

可选方案：

- PNG 序列帧：第一版首选，容易检查和替换
- 透明 WebP：文件较少，但调试单帧不如 PNG 方便
- Sprite Sheet：加载效率较高，但素材制作和切割更复杂

Live2D、Spine、Rive 等方案放到后续版本评估。

## 6. 第一版核心功能清单

### 6.1 窗口功能

- [ ] 背景完全透明
- [ ] 无系统标题栏和边框
- [ ] 默认始终置顶
- [ ] 可在设置中关闭始终置顶
- [ ] 可用鼠标拖动桌宠
- [ ] 保存最后窗口位置
- [ ] 保存桌宠缩放比例
- [ ] 支持多个显示器
- [ ] 显示器配置改变时自动修正位置
- [ ] 不允许桌宠完全移出可见区域
- [ ] 支持临时隐藏和恢复
- [ ] 防止重复启动多个实例
- [ ] 全屏应用运行时可以选择自动隐藏

### 6.2 动画状态

第一版至少支持：

- `idle`：待机
- `blink`：眨眼
- `walk`：走路
- `sleep`：睡觉
- `click`：点击反应
- `drag`：被拖动
- `land`：松开后的落地反应
- `happy`：单次开心动作

状态系统要求：

- [ ] 支持循环动画
- [ ] 支持单次动画
- [ ] 支持动画结束后返回指定状态
- [ ] 每个动作可单独设置 FPS
- [ ] 每个动作可设置播放权重
- [ ] 支持随机等待时间
- [ ] 支持水平翻转
- [ ] 支持动作优先级
- [ ] 支持不可中断动作
- [ ] 支持缺失资源回退
- [ ] 状态切换不能出现空白闪烁

### 6.3 鼠标互动

- [ ] 单击触发可配置动作
- [ ] 双击触发可配置动作
- [ ] 鼠标按下并移动时进入拖动状态
- [ ] 松开后触发落地状态
- [ ] 鼠标悬停时可触发眨眼或轻微反应
- [ ] 点击和拖动有明确阈值，避免误判
- [ ] 互动动作支持冷却时间
- [ ] 可在设置中关闭互动

### 6.4 右键菜单与系统托盘

- [ ] 暂停或继续动画
- [ ] 调整大小
- [ ] 调整透明度
- [ ] 开启或关闭置顶
- [ ] 开启或关闭开机启动
- [ ] 切换角色
- [ ] 切换皮肤
- [ ] 打开设置
- [ ] 打开开发者面板
- [ ] 重新加载角色资源
- [ ] 临时隐藏
- [ ] 恢复默认位置
- [ ] 退出程序

### 6.5 设置保存

本地保存：

- 窗口位置
- 当前显示器
- 缩放比例
- 透明度
- 当前角色
- 当前皮肤
- 置顶状态
- 开机启动状态
- 动画暂停状态
- 音量
- 是否在全屏应用时隐藏
- 是否显示开发者面板

保存失败时应使用默认设置启动，不能阻止程序运行。

## 7. 动画状态机建议

### 7.1 状态优先级示例

由高到低：

1. `drag`
2. `land`
3. `click`
4. `happy`
5. `sleep`
6. `walk`
7. `blink`
8. `idle`

### 7.2 基础转换规则

```text
启动 -> idle
idle -> blink -> idle
idle -> walk -> idle
idle -> sleep
sleep -> click -> idle
任意可中断状态 -> drag -> land -> idle
idle -> click -> idle
idle -> happy -> idle
```

### 7.3 状态配置字段

每个动画至少支持：

```ts
interface AnimationDefinition {
  path: string;
  fps: number;
  loop: boolean;
  returnTo?: string;
  interruptible?: boolean;
  priority?: number;
  weight?: number;
  minDelayMs?: number;
  maxDelayMs?: number;
  offsetX?: number;
  offsetY?: number;
  scale?: number;
  flipXAllowed?: boolean;
}
```

### 8.3 资源包校验

启动或导入角色时检查：

- `manifest.json` 能否解析
- `schemaVersion` 是否受支持
- `id` 是否唯一
- 必需的 `idle` 动画是否存在
- 图片是否为支持的格式
- 同一动作的画布尺寸是否一致
- 文件名是否能正确排序
- 帧率是否在合理范围
- 锚点是否有效
- 路径是否越出角色目录

校验失败时应显示具体错误，不允许只显示“加载失败”。

## 9. 开发者工具

### 9.1 开发者面板

建议提供仅在开发模式默认显示的面板：

- 当前角色 ID
- 当前动画状态
- 当前帧编号
- 当前 FPS
- 图片缓存数量
- 窗口坐标和显示器信息
- 最近一次状态切换原因
- 最近错误日志

- 暂停在指定帧
- 开启或关闭锚点辅助线
- 开启或关闭角色边界框
- 重新加载当前角色

### 9.2 日志

日志至少分为：

- `debug`
- `info`
- `warn`
- `error`

正式版默认不显示详细调试信息，但应保留本地错误日志，方便排查资源加载和窗口问题。

### 9.3 资源检查脚本

提供命令：

```bash
npm run validate:characters
```

脚本应输出：

- 发现的角色数量
- 每个角色的动作数量
- 缺失文件
- 不一致的画布尺寸
- 无法解析的配置
- 重复 ID
- 不连续或无法排序的帧名

## 10. 外观制作的强制确认流程

### 10.1 核心规则

**在生成任何正式桌宠外观、角色设定图、表情图或动作素材之前，必须先询问用户希望生成什么样的风格。**

未确认画风时：

- 不生成正式角色图
- 不批量生成动作帧
- 不擅自选择二次元、写实、像素或其他风格
- 只允许使用通用占位素材继续开发程序

如果用户已经在同一轮中明确提供了完整风格要求，则无需重复询问。

### 10.2 必须先问的问题

最少必须询问：

> 你希望桌宠生成什么样的画风？例如：日系 Q 版、韩系 Q 版、像素风、手绘插画、卡通动物、3D 软萌、赛博朋克或其他自定义风格。

为了提高结果稳定性，建议同时确认：

1. 角色类型：人物、动物、机器人、幻想生物或其他
2. 画风：用户指定
3. 身体比例：2 头身、2.5 头身、3 头身、正常比例等
4. 主色调：暖色、冷色、莫兰迪、鲜艳、黑白等
5. 服装或外壳设计
6. 性格气质：可爱、安静、活泼、酷、搞怪等
7. 是否有参考图片
8. 是否需要保持某个真实人物或已有角色的特征
9. 第一版需要哪些动作
10. 最终展示尺寸和目标平台

### 10.3 风格确认表

在开始生成前，应记录：

```yaml
character_name: "待填写"
character_type: "人物 / 动物 / 机器人 / 幻想生物 / 其他"
art_style: "必须由用户确认"
body_ratio: "待填写"
visual_mood: "待填写"
main_colors:
  - "待填写"
  - "待填写"
outfit_or_shell: "待填写"
hair_or_head_features: "待填写"
accessories:
  - "待填写"
special_features:
  - "待填写"
reference_images: "无 / 有"
first_animation_set:
  - "idle"
  - "blink"
  - "click"
  - "drag"
```

### 10.4 可直接使用的确认话术

```text

4. 手绘卡通

```

## 11. 外观制作流程

确认画风后，按以下顺序进行：

1. 确认角色类型和画风
2. 确认身体比例、主色、服装和标志性特征
3. 确认是否使用参考图片
4. 生成 2～4 个静态概念方向
5. 用户选择一个方向
6. 制作统一的角色设定图
7. 锁定角色设定
8. 制作表情表
9. 制作动作关键姿势
10. 补充连续动画帧
11. 清理透明背景和边缘
12. 按资源包规范命名与导出
13. 导入程序验证
14. 修正比例、锚点、动作节奏和一致性

禁止在角色设定尚未确认时，直接批量制作所有动画。

---

## 12. 是否需要全方位参考图

### 12.1 原创角色

如果是完全原创的桌宠，不需要真人全方位照片。只需要明确：

- 角色类型
- 画风
- 轮廓
- 配色
- 服装或外壳
- 标志性特征
- 身体比例

### 12.2 基于真人的角色

不需要专业 360 度扫描，但多角度、清晰、经过本人授权的参考图片有助于保持一致性。

最低可用：

- 正脸
- 侧脸或 3/4 角度
- 正面全身
- 发型或服装细节

较稳定的参考组：

- 正面半身
- 正面全身
- 左侧或右侧脸
- 3/4 角度
- 发型细节
- 服装正面
- 服装侧面或背面
- 平静表情
- 微笑表情

### 12.3 基于现有角色或品牌形象

制作前应确认：

- 是否拥有使用授权
- 是否允许公开发布
- 是否允许商业使用
- 是否允许修改和制作衍生形象

角色资源包建议附带来源和授权说明。

## 13. 角色设定图生成模板

只有在用户确认风格后，才填写并使用此模板。

```text
根据已经确认的角色需求，设计一个可用于 Windows 桌宠的角色设定图。

已确认信息：
- 角色类型：{character_type}
- 画风：{art_style}
- 身体比例：{body_ratio}
- 主要颜色：{main_colors}
- 服装或外壳：{outfit_or_shell}
- 性格气质：{visual_mood}
- 标志性特征：{special_features}

要求：
1. 保持统一的角色身份、轮廓、配色和比例。
2. 同一张设定图展示正面、侧面、背面和 3/4 角度。
3. 展示头部近景和至少六种基础表情。
4. 线条、材质、光影和细节复杂度符合已确认的画风。
5. 轮廓清晰，缩小到桌宠显示尺寸时仍易识别。
6. 各视角的服装结构、饰品和颜色必须一致。
7. 角色之间留有间距，不重叠。
8. 不添加无关人物、文字、水印或复杂场景。
9. 设计应便于后续拆分为透明动画素材。
10. 未经确认不得改变画风或增加新的关键设计元素。
```

### 13.1 设定图确认标准

- 是否符合用户确认的画风
- 不同视角是否为同一角色
- 轮廓是否统一
- 配色是否统一
- 服装或外壳是否统一
- 比例是否统一
- 标志性特征是否保留
- 缩小后是否仍清晰
- 是否适合制作动画

只有通过确认后，才开始正式动作素材。

## 14. 第一版动作素材规划

| 动作 | 建议帧数 | 是否循环 | 建议帧率 |
|---|---:|---|---:|
| 待机呼吸 | 6～10 | 是 | 6～10 FPS |
| 眨眼 | 3～5 | 否或嵌入待机 | 8～12 FPS |
| 点击反应 | 5～8 | 否 | 10～14 FPS |
| 被拖动 | 1～4 | 持续 | 按状态显示 |
| 落地 | 4～8 | 否 | 10～14 FPS |
| 走路 | 8～12 | 是 | 10～14 FPS |
| 睡觉 | 6～10 | 是 | 5～8 FPS |
| 开心 | 6～10 | 否 | 10～14 FPS |

第一版优先顺序：

1. `idle`
2. `blink`
3. `click`
4. `drag`
5. `land`
6. `walk`
7. `sleep`
8. `happy`

## 15. 动作素材通用生成模板

```text
基于已经确认并锁定的角色设定图，为同一个角色制作 {animation_name} 动画关键帧。

要求：
1. 严格保持已确认的画风，不得改变渲染方式、线条、材质或比例。
2. 严格保持脸部或头部结构、服装或外壳、配色、饰品和标志性特征一致。
3. 每一帧使用相同的画布尺寸和角色基准尺寸。
4. 角色锚点保持稳定，不得在各帧中无规律位移。
5. 首尾帧按照动作需要自然循环或返回待机。
6. 背景透明，完整显示角色，不裁切关键部分。
7. 不添加文字、水印、复杂场景或不必要的特效。
8. 不擅自增加用户未确认的服装、饰品或配色。

10. 输出后需要检查角色一致性、锚点和画布尺寸。
```

## 16. 美术资源技术规范

### 16.1 推荐格式

- 文件格式：PNG
- 背景：透明
- 色彩模式：RGBA
- 单帧画布：建议 `512 × 512` 或 `1024 × 1024`
- 最终显示高度：通常约 `120～300` 像素，由程序缩放
- 同一动作所有帧必须使用相同画布尺寸
- 同一角色不同动作应保持接近的视觉基准尺寸

### 16.2 命名

```text
idle_0001.png
idle_0002.png
idle_0003.png
```

要求：

- 使用小写英文动作名
- 使用固定长度的数字序号
- 不使用空格
- 不混用中文标点
- 文件排序后必须与播放顺序一致

### 16.3 锚点与对齐

- 站立动作统一脚底位置
- 待机、走路和点击尽量使用同一锚点
- 睡觉等特殊姿势允许配置独立偏移
- 程序支持每个动作设置 `offsetX`、`offsetY` 和 `scale`
- 不允许每一帧自动裁剪成不同大小

### 16.4 导入前检查

- 透明边缘无白边或黑边
- 帧尺寸一致
- 角色没有突然放大或缩小
- 服装、外壳和饰品不跳变
- 脚底或中心点不明显抖动
- 首尾帧衔接自然
- 文件名连续且可排序
- 缺失帧有明确记录

## 20. 最终执行原则

当前项目应被视为一个“桌宠运行框架和角色播放器”，而不是某个特定用途的应用。

第一版先做到：

- 能安装
- 能启动
- 能显示
- 能拖动
- 能播放动画
- 能切换状态
- 能加载角色资源包
- 能保存设置
- 能调试和排错
- 能安全回退

正式外观制作必须遵守：

> 先询问用户希望生成什么样的风格，确认后再设计角色；未确认风格时只使用占位素材，不生成正式外观。

这样可以避免过早绑定某种画风，也能让核心程序以后适配人物、动物、像素、卡通、机器人和其他不同类型的桌宠。
